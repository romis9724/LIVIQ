## 1. 프로젝트 개요

### 1.1 프로젝트명
LIVIQ — 아파트 관리 업무 지원 AI 플랫폼

### 1.2 한 줄 정의
관리사무소의 문서·이력 데이터를 검색·요약하고, 입주민의 반복 문의를 1차 응대하여
**사람의 업무를 줄이는 보조 도구**다. 사람을 대체하는 무인 운영 시스템이 아니다.

### 1.3 목적
기존 아파트 관리 업무의 비효율을 LLM·RAG로 줄인다. 우선순위가 높은 순서:
1. 반복 문의 1차 자동 응대 (가장 효과가 크고 위험이 낮음)
2. 관리규약·회의록·공지 문서 검색 자동화
3. 시설 유지보수 이력의 검색·요약(지식 단절 완화)
4. 공지문·알림 초안 자동 작성

### 1.4 비목표 (Non-Goals)
범위를 명확히 하기 위해 **하지 않을 것**을 먼저 정의한다.
- 회계·관리비 부과 계산을 AI가 **직접 수행**하지 않는다. (정합성·법적 책임 문제, 기존 ERP가 담당)
- 법적 효력이 필요한 안내(대피 지시, 규약 유권해석)를 AI 응답으로 **대체**하지 않는다.
- 사람 검수 없이 입주민에게 자동 발송되는 공지를 만들지 않는다. (초안 생성까지만)
- 초기에는 3D 디지털 트윈, BIM 연동을 하지 않는다. (9장 참고)

## 2. 현황 분석

시장은 이미 성숙해 있다. 입주민 편의 기능과 관리 백오피스는 기성 솔루션이
폭넓게 제공하므로, LIVIQ가 같은 기능을 다시 만드는 것은 의미가 없다.
대표 솔루션을 벤치마킹해 **무엇이 이미 표준(table-stakes)인지**를 먼저 못 박는다.
(상세 화면은 [refs/](../refs/) 참고)

### 2.1 벤치마킹 — 입주민 앱 「아파트너」
입주민 생활 편의·소통에 특화. 이미 제공 중인 기능:
- 생활 편의: 민원·하자 접수(처리상태 추적), 방문 차량 예약(주차관제 연동), 커뮤니티센터 예약·정산, 주차 안심번호, 입주·이사 예약, 모바일 카드키(S1), 단지 CCTV, 셔틀/통학버스 실시간 위치
- 정보 조회: 공지 알림, 관리비 조회(추이 그래프), 주요 일정 캘린더, 주변 정보(플레이스)
- 소통: 입주민 게시판, 입대의·선관위 전용 게시판, 동네 게시판, 전문가 상담, 중고마켓, 설문조사, 전자투표
- 관리자: 전자결재, 전자 관리비 고지서, 공문서 수·발신, 안내방송(문자↔음성)

### 2.2 벤치마킹 — 관리 웹 「아파트데스크」
회계·수납 백오피스에 특화. 이미 제공 중인 기능:
- 자동이체: 신청자 조회, 월별 신청자, 이중 신청자 탐지, 신청자 확인 요청
- 수납: 자동이체/기타/이중 수납 조회, 은행 지점 처리, 간편결제(아파트아이·토스·네이버페이 등)
- 카드: 승인 내역, 승인 취소, 전출자 자동이체 해지, 재승인, 부가세 매출자료
- 금액: 자동이체 금액 수정(금융사별 처리 기준 반영)

### 2.3 LIVIQ의 차별화 지점 (위 기능에 AI 레이어를 얹는다)
기능 재구현이 아니라, 기성 기능이 **다루지 못하는 영역**에 집중한다.

| 기존 솔루션이 잘하는 것 | 못 다루는 것 | LIVIQ의 AI 레이어 |
|--------------------------|--------------|--------------------|
| 공지·규약·게시판 "게시" | 흩어진 문서 "검색·요약" | RAG 문서 검색, 출처 인용 응대 (5.1·5.2) |
| 민원 접수·상태 추적 | 반복 민원 1차 "응답" | AI 민원 1차 응대·자동 분류 (5.1) |
| 관리비 금액 "조회/부과" | "왜 올랐는지" 설명 | 확정 데이터 기반 비교·설명 (5.6) |
| 설비 점검·이력 "기록" | 과거 이력 "검색·원인 추정" | 시설 도우미, 유사 장애 검색 (5.4) |
| 회의 일정·게시 | 회의 내용 "요약·액션아이템" | STT + 회의록 요약 (5.5) |
| 공지 "발송" | 공지문 "작성" | 키워드→공지·알림 초안 생성 (5.3) |

> 결론: LIVIQ는 입주민 앱·관리 웹을 다시 만드는 프로젝트가 아니라,
> **기존 시스템/문서 위에서 동작하는 AI 검색·응대·요약 계층**이다.
> 가능하면 기성 앱과 연동(또는 그 데이터 위에 구축)하고, AI 기능에 집중한다.

### 2.4 데이터 현실 (구축 전 반드시 확인)
계획의 성패는 데이터 상태에서 갈린다. 착수 전 다음을 점검한다.
- 관리규약·회의록·공지가 **검색 가능한 텍스트인지**, 스캔 이미지/HWP인지
- 유지보수 이력이 **구조화 DB**인지, 자유 서술 메모인지
- 기존 ERP가 **조회 API/DB 접근**을 제공하는지, 화면만 있는지
- 단지별로 문서 양식·용어가 얼마나 다른지 (규약은 단지마다 다름)

> 위 항목 중 하나라도 "이미지/메모/접근 불가"라면, 해당 기능은 일정·비용이
> 수배 늘어난다. OCR·정제·구조화가 별도 작업이 된다.

## 5. 핵심 AI 기능 (우선순위·위험도 표기)

각 기능에 **MVP 포함 여부**와 **오답 시 위험도**를 명시한다.
위험도가 높을수록 사람 검수(human-in-the-loop) 비중을 키운다.

### 5.1 AI 민원 1차 응대 — MVP 핵심, 위험 낮음
- 입력 예: "주차 등록 어떻게 해?", "반려견 규정 알려줘"
- 동작: FAQ·규약·공지 RAG 검색 → 출처 인용과 함께 답변 → 모호하면 담당자 연결
- 안전장치: 답변에 **근거 문서·조항을 항상 인용**, 신뢰도 낮으면 "담당자 확인 필요" 안내

### 5.2 AI 문서 검색 — MVP 핵심, 위험 낮음
- 대상: 관리규약, 회의록, 공지, 운영 지침
- 동작: 자연어 질의 → 관련 조항 발췌 + 원문 링크 제공
- 원칙: AI는 **요약·발췌**만, 유권해석은 하지 않음 (해석은 사람)

### 5.3 AI 공지 초안 작성 — MVP 포함, 위험 중간
- 입력: 단수/작업 시간/영향 세대 등 키워드
- 출력: 공지문·알림 **초안** (자동 발송 금지, 관리자 검수 후 발송)

### 5.4 AI 시설관리 도우미 — Phase 2, 위험 중간
- 동작: 과거 장애·유지보수 이력·매뉴얼 검색 → 가능한 원인 **후보 제시**
- 한계: 진단·조치 결정은 사람. 이력 데이터가 부실하면 효용 제한적

### 5.5 AI 회의록 요약 — Phase 2, 위험 중간
- STT 변환 → 요약·결정사항·액션아이템 추출
- 한계: STT 정확도(다화자·방언·소음)에 좌우. **요약은 검수 후 회의록으로 확정**

### 5.6 AI 관리비 설명 — Phase 2, 위험 중간
- 동작: 확정 관리비 데이터(관리자 엑셀 업로드, 추후 ERP 병행)를 **읽어와** 전월·평균 대비 설명
- 원칙: AI는 **계산하지 않고 설명만**. 금액 산출은 확정 업로드 데이터가 단일 출처(source of truth)

### 5.7 (검토) 감정 분석 기반 민원 우선순위 — 보류
- 감정 분석은 한국어 짧은 민원에서 정확도가 낮고 오분류 리스크가 크다.
- 초기에는 **키워드 기반 긴급도 분류 + 반복 민원 탐지**로 대체하고,
  감정 분석은 데이터 축적 후 정확도가 확인되면 도입한다.

## 6. 차별화 기능 (실현 가능성 검토 포함)

### 6.1 시공사 하자보수 연동 — Phase 2, 외부 의존 큼
- 기대: 하자 유형 자동 분류 → 시공사 양식 변환 → 접수 전달
- 현실 제약: **시공사별 접수 채널·양식이 표준화돼 있지 않음**. 자동 전달은
  파트너십·API 협의가 선행되어야 하며, 협의 없이는 "분류 + 변환된 양식 다운로드"까지만 가능.

### 6.2 인수인계 브리핑 — Phase 2
- 기대: 직원 변경 시 유지보수 이력·특이 민원·장애 패턴 요약
- 전제: 평소 기록이 쌓여 있어야 함. 기록 입력 습관이 선행 조건.

### 6.3 개인정보 주의
- 민원·인수인계 요약에는 세대·개인 식별 정보가 섞일 수 있다.
- 입주민 노출 화면에서는 **개인정보 마스킹**을 기본 적용한다. (7장 참고)

## 7. 제약 조건 (법규·개인정보·보안)

> 이 장이 빠지면 계획은 현실성을 잃는다. 착수 전 법무·보안 검토 필수.

### 7.1 법규
- **공동주택관리법**: 관리규약·회의록·관리비 공개 의무. AI 답변이 공식 고지를
  대체할 수 없으며, 공식 게시는 기존 절차를 따른다.
- **개인정보보호법**: 입주민 정보 처리 동의·목적 제한·보관기간 준수.
- **소방시설법·화재안전기준**: 대피 안내는 인증된 소방 설비의 책임 영역.
  AI 경로 안내는 **참고 정보**일 뿐 법적 대피 지시가 아니다. (9장)

### 7.2 개인정보·데이터 거버넌스
- 외부 LLM API로 **개인정보·민감정보를 그대로 전송하지 않는다.** 전처리 단계에서
  이름·동호수·연락처를 마스킹/가명화한다.
- 데이터 보관 위치·접근 권한·로그를 정의하고, 단지/입주민별 데이터를 분리한다.
- 외부 LLM 사용 시 **학습 비사용(opt-out)·데이터 미보관** 약관을 확인·계약한다.

### 7.3 보안
- API 키·시크릿은 코드에 하드코딩하지 않고 시크릿 매니저로 관리.
- ERP 연동은 최소 권한 읽기 계정. 입력값 검증·인젝션 방지.
- 입주민 인증·인가, 요청 레이트 리밋, 감사 로그.

## 8. 기술 아키텍처

### 8.1 구성도
```text
[입주민 앱 / 관리자 웹]
        │
   [API Gateway]  ── 인증/인가, 레이트 리밋, 감사 로그
        │
   [오케스트레이션 서비스]
        ├─ 의도 분류(검색형 / 액션형 / 사람연결)
        ├─ 개인정보 마스킹 전처리
        └─ 응답 후처리(출처 인용, 신뢰도 판정)
        │
   ┌────┴──────────────┐
   │                   │
[RAG 파이프라인]   [기존 ERP (읽기 전용)]
   │                회계/민원/시설 DB
[Vector DB]
 pgvector(우선) → 규모 커지면 Qdrant
   │
[LLM]
 외부 API(Claude/GPT) + 필요 시 sLLM
```

### 8.2 기술 선택과 근거
| 영역 | 선택 | 근거 |
|------|------|------|
| Vector DB | **pgvector 우선** | 기존 PostgreSQL 재사용, 운영 단순. 데이터 급증 시 Qdrant 이전 |
| LLM | 외부 API 기본 | 초기 품질·속도 우선. 비용/개인정보 이슈 영역만 sLLM 검토 |
| RAG | 청킹 + 임베딩 + **리랭킹** + 출처 인용 | 인용 없는 답변은 신뢰 불가 |
| 평가 | 골든셋 기반 자동 평가 + 표본 수동 채점 | 환각률·정답률을 수치로 관리 |

> "GPT/Claude/sLLM 하이브리드"는 멋진 말이지만, 하이브리드는 운영 복잡도를 키운다.
> **초기에는 외부 API 단일 경로로 시작**하고, 비용·개인정보 데이터가 쌓인 뒤
> sLLM 도입 ROI를 재평가한다. (조기 최적화 회피)

### 8.3 LLM 한계와 대응 (반드시 설계에 반영)
- **환각**: 모든 답변에 출처를 강제하고, 근거를 못 찾으면 "모름 + 담당자 연결"로 폴백.
- **권한 외 응답 차단**: 시스템 프롬프트만으로 막지 않고, 액션은 별도 권한 검증 계층에서 통제.
- **품질 모니터링**: 응답 로그·👎 피드백을 수집해 골든셋을 갱신하고 회귀 평가.

## 9. 2D 운영형 시설 시각화 (디지털 트윈은 후순위)

### 9.1 방향
초기에는 3D 디지털 트윈이 아니라 **2D 도면 기반 설비 상태 시각화**로 시작한다.
"디지털 트윈"이라는 용어는 기대치를 과도하게 높이므로, MVP 단계에서는
**"설비 상태 대시보드"**로 명명한다.

### 9.2 2D부터 시작하는 이유
- 3D/BIM: 모델링 비용·BIM 데이터 부족·유지보수 부담 대비 초기 ROI 낮음.
- 2D: 기존 평면도(PDF/CAD) 활용, 빠른 구축, 관리사무소 적응 용이.

### 9.3 MVP 범위 / 제외
- **포함**: 기계실·전기실·급수/배수 펌프·공용 전력의 2D 상태 표시(정상/점검중/장애/위험)
- **제외(초기)**: 전체 3D 모델, 고정밀 BIM, 실시간 3D 렌더링

### 9.4 핵심 기능 (난이도순)
- **A. 설비 상태 맵**: 2D 평면에 설비 상태 색상 표시 — *난이도 낮음*
- **B. 자연어 설비 검색**: "점검 기한 지난 설비 보여줘" → 조회·위치 표시 — *낮음*
- **C. 위험 설비 요약**: "오늘 점검 필요 설비 요약" → 우선순위 안내 — *중간*
- **D. 규칙 기반 이상 탐지**: 임계값 초과(압력/온도/전력) 알림 — *중간*
  - 초기에는 ML 예측이 아니라 **임계값·추세 규칙**으로 시작 (데이터·정답이 없으면 ML은 과적합/오탐)
- **E. (R&D) 재난 시 대피 경로 참고 표시**: 화재·연기 센서 연동 경로 안내 — *높음, Phase 3*

### 9.5 대피 경로 기능에 대한 현실적 입장
- 이 기능은 **생명 안전(life-safety) 영역**으로, 센서 신뢰성·실시간성·인증이 핵심이다.
- 법적 대피 지시는 **인증된 소방 설비**의 역할이며, LIVIQ 경로 안내는 **보조 참고 정보**로만 제공한다.
- 따라서 MVP가 아닌 **Phase 3 R&D 과제**로 분리하고, 소방·안전 전문가 검토와
  실패 모드(센서 오작동·정전·통신 두절) 설계를 전제로 한다. A* 등 경로 탐색은
  그 다음 문제다.

### 9.6 도면 데이터 파이프라인
1. **정제**: 원본 CAD(DXF/DWG)에서 벽·문·계단 등 필수 레이어만 추출
2. **변환**: 웹 엔진이 읽는 SVG/GeoJSON으로 변환
3. **태깅**: 구역·설비에 고유 ID 부여 → 센서·AI 연동
> 단지마다 도면 품질·레이어 규칙이 다르므로 **단지당 초기 셋업 비용**이 발생한다.

### 9.7 데이터 구성
- 실시간: 압력·온도·전력·설비/점검 상태 (+ 안전 단계에서 화재/연기 센서)
- 운영: 유지보수·장애 이력, 점검 로그, 매뉴얼
- 공간: SVG/GeoJSON 노드·엣지(경로용)

### 9.8 기술 스택 (확정 아님, 후보)
- 수집: MQTT, (대량 시) Kafka, Redis Stream
- 시계열: TimescaleDB 또는 InfluxDB
- 2D 시각화: SVG 기반 UI, Konva.js/React Flow (요구사항 확정 후 택1)
- 경로 탐색(Phase 3): A*/Dijkstra
- AI: LangChain/LlamaIndex 중 택1 + 외부 LLM
> IoT/BMS 연동은 단지 설비·통신 규격 실사 후 확정한다.

## 10. 단계별 로드맵 (개략, 파일럿 후 보정)

기간은 인력·데이터 상태에 따라 변동하는 **가정치**다.

| 단계 | 기간(가정) | 범위 | 산출물 / 종료 기준 |
|------|-----------|------|--------------------|
| 0. 준비 | 2~3주 | 데이터 실사, 법무·보안 검토, 골든셋 구축 | 데이터 현황 보고서, 평가셋 |
| 1. MVP | 6~10주 | 민원 1차 응대 + 문서 검색 + 공지 초안 (5.1~5.3) | 1개 단지 파일럿, KPI 측정 |
| 2. 확장 | 2~3개월 | 시설 도우미·회의록 요약·관리비 설명, 2D 상태맵 A~C | 시설팀 사용, 다단지 확대 |
| 3. 고도화 | 추후 | 이상 탐지 ML, 하자 연동, (R&D) 대피 경로 | ROI 재평가 후 결정 |

## 11. 비용 구조 (검증 전 가정)

정확한 견적이 아니라 **비용을 좌우하는 변수**를 정의한다.

- **LLM API**: 질의량 × (입력 토큰 + 출력 토큰) 단가. RAG는 컨텍스트가 길어
  질의당 토큰이 큰 편 → **캐싱·청크 길이·리랭킹**으로 비용을 통제한다.
- **임베딩/인덱싱**: 문서량에 비례(초기 일괄 + 증분).
- **인프라**: Vector DB, 시계열 DB, 앱/오케스트레이션 서버.
- **초기 셋업(단지당)**: 문서 정제·OCR, 도면 변환·태깅 — **일회성·노동 집약적**.
- **운영**: 품질 모니터링·골든셋 갱신·사람 검수 인력.

> 파일럿 1개 단지에서 **질의당 실제 토큰·비용**을 측정해 단지 확대 시 단가를 산정한다.

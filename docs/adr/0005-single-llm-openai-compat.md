# ADR-0005: 단일 LLM + OpenAI-호환 엔드포인트 추상화

- 상태: Accepted
- 날짜: 2026-07-13
- 관련: [docs/01 §12 ADR-2](../01-architecture.md), [docs/08 §4](../08-llm-token-optimization.md), [ADR-0002](0002-mask-before-external-llm.md)
- 갱신: 2026-07-13 — 파일럿 기본 엔드포인트 = **Ollama 로컬**(생성 + 임베딩 bge-m3). 품질 미달 시 골든셋 비교로 외부 API 승격.

## 맥락

초기 설계는 "외부 LLM API(Claude 우선) 단일 경로"였다. 운영 방침이 바뀌어
Ollama·vLLM(self-hosted)·OpenAI-호환 API를 **성능비교 후 선택**해 쓰려 한다.
단, 동시에 여러 모델을 굴리는 멀티 라우팅은 원하지 않는다 — 비교를 위한 교체 용이성이 목적.

## 결정

**생성 모델은 동시에 하나만 운영**하고, `ai-core/llm`은 **OpenAI-호환 클라이언트 1개**로 추상화한다.
프로바이더 교체는 env(`LLM_BASE_URL`, `LLM_MODEL`, `LLM_API_KEY`)만으로 가능해야 한다.
모델 선정·교체는 골든셋 회귀 평가([docs/07](../07-testing-strategy.md))의 정량 비교(적중률·환각률·비용)로 결정한다.
임베딩은 생성 모델과 별개로 **bge-m3(1024, cosine)** 고정 — pgvector·Neo4j 벡터 차원의 기준이므로
교체는 전량 재색인 이벤트로만 허용한다.

마스킹 정책은 [ADR-0002](0002-mask-before-external-llm.md)를 **전 프로바이더로 확대 적용**한다:
self-hosted(Ollama·vLLM)도 예외 없이 호출 전 마스킹(fail-closed). 0002의 재검토 신호
("온프레미스 LLM 도입 시")가 발생했고, 재검토 결과 **유지·강화**로 결론 — 프로바이더별로
정책이 갈리면 모델 교체 때마다 보안 재검토가 필요해져 교체 용이성이라는 본 결정의 목적과 상충한다.

## 대안

- **특정 벤더 SDK 고정(Claude 등)**: 품질은 예측 가능하나 벤더 락인, 성능비교 불가. 기각.
- **멀티 모델 작업별 라우팅**: 토큰 절약 여지는 있으나 운영·평가 복잡도 2배 이상,
  현 단계 목적(단일 모델 선정)과 불일치. **보류** — 필요가 검증되면 재도입.
- **LiteLLM 등 게이트웨이 도입**: 셋 다 OpenAI-호환이라 게이트웨이 없이 baseURL 교체로 충분. YAGNI. 기각.
- **self-hosted에 원문 전송 허용**: 품질 이득 대비 서버 위치·로그 통제 책임 발생, 정책 이원화. 기각.

## 결과

- `ai-core/llm`은 OpenAI-호환 인터페이스 하나만 구현·테스트하면 된다(프로바이더별 어댑터 불필요).
- 모델 비교 절차 = 같은 골든셋을 env만 바꿔 재실행 → 리포트 비교. 평가 하네스가 사실상 벤치 도구.
- self-hosted 비용 축은 API 과금이 아닌 토큰 수+지연(GPU 자원)으로 동일 기록([docs/08 §9](../08-llm-token-optimization.md)).
- 재검토 신호: 단일 모델로 품질·비용 목표를 동시 충족 못 해 작업별 분리가 필요해질 때(멀티 라우팅 재논의).

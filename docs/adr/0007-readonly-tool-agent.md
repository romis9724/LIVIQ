# ADR-0007: 읽기 전용 도구호출 에이전트 (정적 라우터 대체)

- 상태: Accepted
- 날짜: 2026-07-13
- 관련: [docs/01 §5.2](../01-architecture.md), [docs/01 §12 ADR-9](../01-architecture.md), [ADR-0005](0005-single-llm-openai-compat.md)

## 맥락

기존 질의 파이프라인은 **정적 라우터**였다 — 의도분류가 경로 1개(검색형/시설형/액션형)를 골라 고정 파이프라인으로 흘려보냈다. 그러나 데이터가 여러 스토어로 이원화(pgvector 문서 + Neo4j 시설)되고 정형 SQL 조회(관리비·민원·설비)까지 얹히면서, 하나의 질의가 여러 소스를 동시에 필요로 하는 **복합 질의**를 정적 라우터로는 커버할 수 없다. 예: "관리비가 왜 올랐고 엘리베이터 점검은 언제인가?" — `get_fees`와 `get_facilities`를 함께 호출해야 답이 나온다.

제약: 토큰은 1급 비용([08](../08-llm-token-optimization.md))이라 자유 루프는 위험하고, 규칙 8("액션은 코드가 실행")상 LLM이 부수효과를 직접 트리거해선 안 된다.

## 결정

질의 처리를 **읽기 전용 도구 레지스트리 + LLM 도구호출 에이전트**로 개편한다. LLM은 도구 레지스트리를 보고 필요한 도구를 선택·조합해 호출하고, **스텝 상한 2~3회**(질의당 도구 호출 수·토큰 예산)를 둔다 — 초과 시 현재 근거로 답변하거나 폴백한다.

- 도구는 **읽기 전용만**: `search_documents`(pgvector) · `search_facility_graph`(Neo4j) · 고정 SQL 4종(`get_fees`·`get_my_inquiries`·`get_facilities`·`get_overdue_checks`). 도구 표 전문은 [01 §5.2](../01-architecture.md).
- **파라미터 Zod 검증·tenant·소유권·읽기전용 강제는 코드**가 담당한다. LLM은 도구 선택만 하고, 실행은 검증된 코드 경로를 통과한다 — 규칙 8과 충돌 없음.
- 도구 결과도 문서 인용과 동일하게 **출처 카드**로 표시한다(예: "근거: 관리비 2026-06 확정 데이터").
- **경로 로깅·골든셋 평가**: 어떤 도구를 왜 호출했는지 기록하고, 골든셋에 "기대 도구 경로"를 추가해 회귀 평가한다([07 §5](../07-testing-strategy.md)).
- 캐시(정확/의미)는 에이전트 **앞단**에서 먼저 확인하고([08](../08-llm-token-optimization.md)), 히트면 에이전트에 진입하지 않는다. 의도분류는 유지하되 역할을 "AI 처리 vs 사람연결" 1차 분기로 축소 — 데이터 소스 선택은 에이전트가 한다.

## 대안

- **정적 라우터 유지**: 단순하나 복합 질의를 커버 못 하고, 소스가 늘 때마다 라우팅 분기를 손봐야 한다. 기각.
- **자유 ReAct 루프(무제한 스텝)**: 유연하나 토큰·지연·평가를 통제할 수 없다(루프 폭주=최대 토큰 리스크). 스텝 상한으로 절충. 기각.
- **쓰기 도구 포함(발송·권한 변경 등)**: LLM이 부수효과를 트리거하게 되어 규칙 8 위반. 쓰기는 UI/폼으로 유지. 기각.

## 결과

- 복합 질의를 커버하고, 새 소스는 **도구 추가만으로 확장**된다(파이프라인 스키마 변경 불필요).
- 비용은 스텝 상한·도구 결과 크기 상한으로 통제([08 §8](../08-llm-token-optimization.md)).
- 평가 트랙에 "도구 경로 적정성" 지표가 추가된다([07 §5](../07-testing-strategy.md)).
- 재검토 신호: 스텝 상한 2~3회로 답을 못 내는 복합 질의 사례가 누적되면 상한·루프 전략을 재논의한다.

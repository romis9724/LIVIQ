# ADR-0012: MVP 알림은 인앱 알림함만, 웹푸시는 Phase 2

- 상태: Accepted
- 날짜: 2026-07-13
- 관련: [docs/04](../04-menu-structure.md), [docs/05](../05-ui-ux-design.md), [docs/03 §6](../03-database-design.md), [docs/09 §8](../09-implementation-harness.md)

## 맥락

공지 발행·AI 검수 결과·민원 상태 변경을 사용자에게 알릴 채널이 필요하다. 알림이 전혀 없으면 입주민이 앱을 직접 열어야만 확인 가능해 **공지 전달 가치가 훼손**되고, 검수 게이트(규칙 6) 흐름에서 검수 결과를 담당자에게 통지할 수단이 없다. 반면 브라우저 웹푸시는 iOS PWA 제약(홈 화면 추가 + iOS 16.4+에서만 동작), 구독 인프라(VAPID 키·구독 저장·재구독·만료 처리), 발송 신뢰성(재시도·실패 추적) 부담이 있어 MVP 범위를 크게 키운다.

## 결정

MVP는 **인앱 알림함(`notifications` 테이블 + 배지·목록 UI)만** 제공한다. 웹푸시는 **Phase 2**로 미룬다.

- 알림은 도메인 이벤트(공지 발행·검수 완료·민원 상태 변경)에서 **서버가 `notifications`에 기록**한다.
- 클라이언트는 재방문·폴링으로 조회한다(실시간 서버 푸시 없음).
- `notifications`는 tenant·user 소유 데이터이므로 RLS 적용, 개인정보는 참조/마스킹 원칙 준수([docs/03 §6]).

## 대안

- **웹푸시 동시 도입**: 도달률은 오르나 iOS PWA 제약으로 대상 일부만 수신하고, 구독 인프라·발송 재시도·권한 요청 UX가 MVP 범위를 확대한다. 파일럿 접속 빈도 데이터 없이는 투자 대비 효용을 판단할 수 없다. 보류.
- **알림 채널 없음(화면에서만 확인)**: 인프라 비용은 0이나 공지·검수 결과 전달 가치가 훼손된다(규칙 6 검수 흐름 통지 불가). 기각.

## 결과

- 발송 인프라(VAPID·구독 관리·푸시 게이트웨이) 없이 테이블 + 조회 API + 배지 UI만으로 완성 — MVP 범위 최소.
- 실시간성은 재방문·폴링 수준(즉시 푸시 아님) — 파일럿에서 수용 가능한 것으로 가정.
- 재검토 신호: 파일럿 **접속 빈도 데이터**로 재방문만으로는 공지 도달이 부족하다고 확인되면 웹푸시(Phase 2)를 착수한다.

# ADR-0010: pii_vault 봉투 암호화 — env 마스터 키(앱 레벨), KMS는 확장 시 승격

- 상태: Accepted
- 날짜: 2026-07-13
- 관련: [docs/06 §4.1](../06-security-privacy.md), [docs/03 §6](../03-database-design.md), [docs/09 §7](../09-implementation-harness.md), [ADR-0002](0002-mask-before-external-llm.md)

## 맥락

식별정보는 `pii_vault`에 **암호화** 저장하고 업무 테이블은 참조키만 둔다([docs/03 §6], [docs/06 §4.1]). 보안 설계는 "KMS 기반 envelope(키 버전 관리·회전)"를 목표로 명시했으나, 파일럿 인프라(클라우드 프로바이더·관리형 KMS)가 아직 확정되지 않았다. 봉투 암호화의 **마스터 키(KEK)를 어디에 두고 어떻게 공급할지**를 지금 결정하지 않으면 첫 마이그레이션(암복호 경로)을 짤 수 없다. 파일럿 규모는 단일 단지(90세대)라 관리형 KMS의 운영·비용 이점이 아직 크지 않다.

## 결정

**봉투 암호화 구조(KEK로 데이터 키 DEK를 감싸고, DEK로 레코드를 암호화)는 처음부터 채택**하되, 마스터 키(KEK)는 **env `PII_MASTER_KEY`(32byte base64)로 공급하고 봉투 암복호는 앱 레벨**에서 수행한다. 클라우드 KMS로의 승격은 인프라 확정·다단지 확장 시점으로 미룬다 — 봉투 구조를 미리 두었으므로 **KEK 공급원(env → KMS)만 교체**하면 되고 저장 포맷·암복호 경로는 유지된다.

## 대안

- **AWS(또는 GCP) KMS 즉시 도입**: 키 관리 위임·감사·자동 회전 이점이 크나, 파일럿 인프라가 미확정인 상태에서 특정 클라우드에 조기 결합한다. 봉투 구조만 먼저 두면 나중에 무리 없이 승격 가능하므로 지금은 보류. 기각(보류).
- **Vault(self-hosted) 도입**: 키 회전·감사가 강력하나 단일 단지 파일럿에 별도 시크릿 인프라 운영 부담이 과다. YAGNI. 기각.
- **봉투 없이 단일 키 컬럼 암호화(pgcrypto 등)**: 구현은 단순하나 KEK/DEK 계층이 없어 키 회전이 전량 재암호화가 되고, KMS 승격 시 저장 포맷을 갈아야 한다. 기각.

## 결과

- `PII_MASTER_KEY` **유실 = pii_vault 복호 불능**. 백업·복구 절차는 [docs/09 §7](../09-implementation-harness.md)이 소유(마스터 키는 시크릿 매니저 + 오프라인 백업, 데이터 백업과 분리 보관).
- 키 회전은 DEK 재봉투(KEK로 다시 감싸기)로 수행 → 레코드 전량 재암호화 불필요.
- 마스킹 정책([ADR-0002])은 저장 암호화와 **직교** — 봉투 암호화는 at-rest, 마스킹은 LLM 전송 경계. 둘 다 적용.
- 재검토 신호: **다단지 확장** 또는 **컴플라이언스(관리형 키·감사 로그) 요구** 발생 시 KEK 공급원을 KMS로 승격(저장 포맷 불변).

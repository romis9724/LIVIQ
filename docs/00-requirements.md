# 00. 요구사항 정의서 (SRS)

> 상위 기획: [../README.md](../README.md) · 문서 인덱스: [README.md](README.md)
> 본 문서는 LIVIQ가 **무엇을** 해야 하는지를 정의한다. **어떻게**는 [01-architecture.md](01-architecture.md) 이하에서 다룬다.

## 1. 목적과 범위

LIVIQ는 기존 아파트 관리 시스템·문서 위에 얹는 **AI 검색·응대·요약 계층**이다.
MVP는 **입주민 반응형 웹/PWA + 관리자 웹 + AI 코어**를 동시에 구축한다.

- **In Scope (MVP)**: 문서 RAG 검색, 민원 1차 AI 응대, 공지 초안 생성, 관리비 설명, 시설 이력 검색, 회의록 요약, 입주민/관리자 기본 운영 화면
- **Out of Scope (MVP)**: 회계 부과 계산(기존 ERP), 결제/수납 처리, 3D 디지털 트윈/BIM, 재난 대피 경로(9장 R&D), 네이티브 앱
- **비목표**는 [../README.md](../README.md) §1.4를 따른다.

## 2. 사용자 및 역할 (RBAC 기준)

| 역할 | 코드 | 설명 | 주요 권한 |
|------|------|------|-----------|
| 입주민 | `RESIDENT` | 인증된 세대 구성원 | 본인 세대 정보, 공개 문서 검색, 민원 등록, AI 질의 |
| 관리소장 | `MANAGER` | 관리사무소 책임자 | 단지 전체 운영, 공지 발송, 민원 배정, 검수 |
| 관리직원 | `STAFF` | 관리사무소 직원 | 민원 처리, 문서 업로드, 공지 초안 |
| 시설담당 | `FACILITY` | 설비 유지보수 담당 | 시설 이력 검색·등록, 시설 AI 도우미 |
| 입대의/선관위 | `COUNCIL` | 입주자대표회의·선거관리위원회 | 전용 게시판, 회의록, 안건 |
| 시스템관리자 | `SYS_ADMIN` | LIVIQ 운영자 | 테넌트 관리, 모니터링, 골든셋 관리 (단지 데이터 직접 열람 불가, §보안) |

> 멀티테넌시: 모든 역할은 특정 **단지(tenant)** 에 귀속된다. `SYS_ADMIN` 제외.

## 3. 기능 요구사항 (FR)

우선순위 = MoSCoW (M=Must, S=Should, C=Could). 위험도 = 오답·오작동 시 영향.

## 5. 제약 (Constraints)

- **법규**: 공동주택관리법(규약·회의록·관리비 공개 의무, AI는 공식 고지 대체 불가), 개인정보보호법(동의·목적제한·보관), 소방시설법(대피 안내는 인증 설비 책임 — MVP 제외).
- **데이터**: 외부 LLM에 개인정보 미전송, 학습 비사용·미보관 계약 필수.
- **단일 출처**: 관리비 금액은 ERP가 source of truth. AI는 설명만.
- **사람 검수**: 입주민 자동 발송 공지 금지. 위험도 높은 AI 출력은 검수 게이트 통과 필수.

## 6. 가정 및 의존성

- 파일럿 단지 1곳 이상, 세대 정보·문서 제공.
- 기존 ERP가 관리비·세대 데이터를 **읽기 전용**으로 제공(API/DB/파일).
- 관리규약 등 핵심 문서가 전자문서로 확보 가능(아니면 OCR 비용 발생).
- 외부 LLM(API) 사용 승인.

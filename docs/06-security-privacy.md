# 06. 보안 / 개인정보 설계

> DB: [03-database-design.md](03-database-design.md) · 요구사항: [00 §5](00-requirements.md)
> 원칙: **개인정보는 경계에서 차단, 권한은 서버에서 강제, 단지는 격리.**

## 1. 위협 모델 (요약)

| 자산 | 위협 | 대응 |
|------|------|------|
| 입주민 개인정보(이름·연락처·동호수) | 외부 LLM 유출, 로그 노출, 타 입주민 열람 | 마스킹·분리저장·RLS·로그 마스킹 |
| 단지 문서·민원 | 단지 간 혼입, 무권한 접근 | RLS + API 인가 + 공개범위 |
| 관리비 데이터 | 위변조, 무권한 조회 | ERP 읽기전용·미러, 본인 세대 한정 |
| 인증/세션 | 탈취, 권한상승 | 짧은 세션·재발급, 서버 인가, 감사 |
| AI 입력 | 프롬프트 인젝션, 무권한 액션 유도 | 액션=코드 실행, 입력 분리, 권한 재검증 |
| 파일 업로드 | 악성파일, 경로조작 | 타입·크기 검증, 격리 저장, 키 무작위 |

## 2. 인증 / 인가

- **인증**: 로그인 + (선택) 본인확인(PASS). 세대 검증으로 RESIDENT 권한 부여.
- **세션**: 짧은 만료 + 재발급, httpOnly·Secure·SameSite 쿠키. 토큰 사용 시 최소 클레임.
- **인가(RBAC)**: NestJS `RolesGuard` + `TenantGuard`. **프론트 메뉴 숨김은 보조일 뿐**, 모든 엔드포인트가 서버에서 역할·테넌트·소유권을 검증.
- **소유권 검증**: 입주민은 본인/본인 세대 리소스만. (예: `inquiry.author_user_id === me` 또는 같은 household)
- **최소 권한**: ERP 연동은 읽기 전용 계정. 서비스 계정 분리.

## 3. 멀티테넌시 격리 (이중 방어)

1. **애플리케이션**: 모든 쿼리에 `tenant_id` 필터 + 요청 컨텍스트 검증.
2. **DB(RLS)**: 트랜잭션마다 `SET LOCAL app.tenant_id` → 정책으로 강제([03 §5]).
- 벡터 검색도 tenant 선필터(문서 혼입 차단).
- `SYS_ADMIN`은 단지 콘텐츠 비열람. 부득이한 열람은 사전 승인 + 감사 기록.

## 4. 개인정보 보호 (핵심)

### 4.1 데이터 최소화·분리
- 식별정보는 `pii_vault`에 **암호화** 저장, 업무 테이블은 참조키만([03 §6]).
- 검색은 salted 해시. 평문 인덱싱 금지.

### 4.2 외부 LLM 전송 차단 (FR-AI-05, NFR-SEC-01)
- LLM/임베딩 호출 **직전** 마스킹/가명화: 이름→`[이름]`, 동호수→`[세대]`, 연락처→`[연락처]`.
- 마스킹 실패 시 호출 **중단**(fail-closed).
- 외부 LLM 계약: **학습 비사용·데이터 미보관(zero-retention)** 확인.
- 전송 페이로드는 별도 검사 로그(개인정보 패턴 탐지)로 사후 감사.

### 4.3 표시·로그
- 입주민 노출 화면은 마스킹 뷰(`홍*동`, `010-****-1234`).
- 앱 로그·`audit_logs`·트레이스에 개인정보 비저장(마스킹·해시).

### 4.4 동의·보관·파기
- 목적별 동의(`consents`), 철회 가능. 정책 버전 기록.
- 보관기간 만료·탈퇴 시 파기/비식별 배치. 법정 보존 의무 항목은 별도 분리 보관.
- 처리방침 공개, 열람·정정·삭제 요청 처리 절차.

### 4.5 법규 준수
- 개인정보보호법(동의·목적제한·안전조치), 공동주택관리법(규약·회의록·관리비 공개는 **공식 채널**이 수행, AI는 보조).

## 5. 입력 검증 / 인젝션 방지

- 모든 경계 입력 Zod 검증(타입·길이·형식). 신뢰하지 않는 외부 데이터(ERP·LLM 응답) 포함.
- SQL: ORM 파라미터 바인딩(문자열 결합 금지).
- XSS: React 기본 이스케이프, `dangerouslySetInnerHTML` 지양. 사용자 HTML 필요 시 검증된 새니타이저.
- **프롬프트 인젝션**: 검색 컨텍스트와 사용자 지시를 구분(역할 분리), LLM 출력으로 권한·발송을 트리거하지 않음(액션은 코드가 실행, [01 §6]).
- 파일 업로드: 화이트리스트 MIME·확장자·크기, 무작위 저장키, 실행권한 없는 버킷, 안티바이러스(가능 시).

## 6. 전송/저장 보안 · HTTP 헤더

```text
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
Content-Security-Policy: 기본 self, nonce 기반 script, 외부 origin 최소화
```
- TLS 1.2+. 저장 시 민감정보 암호화(at-rest), 키는 시크릿 매니저.
- CSRF: 상태변경 요청 토큰/SameSite. 폼·발송 엔드포인트 레이트 리밋.

## 7. 시크릿 관리

- 코드·레포에 시크릿 금지. `.env`(로컬)·시크릿 매니저(운영).
- 부팅 시 필수 시크릿 존재 검증(Zod), 누락 시 기동 실패.
- 노출 의심 시 즉시 회수·교체. CI에 시크릿 스캐너.

## 8. 감사 / 모니터링

- `audit_logs`(append-only): 로그인, 권한변경, 문서 공개범위 변경, 공지 발송, AI 검수 승인, 개인정보 열람, ERP 동기화.
- 이상 징후: 대량 조회, 비정상 시간대 접근, 마스킹 우회 시도 알림.

## 9. 보안 점검 체크리스트 (배포 전 게이트)

- [ ] 모든 엔드포인트 역할·테넌트·소유권 인가
- [ ] RLS 정책 전 업무 테이블 적용·테스트
- [ ] LLM 전송 페이로드 개인정보 0건(자동 검사)
- [ ] 개인정보 저장 암호화 + 마스킹 뷰 사용
- [ ] 입력 Zod 검증 + 파라미터 쿼리
- [ ] 보안 헤더·CSP·HSTS 적용
- [ ] 시크릿 하드코딩 0, 스캐너 통과
- [ ] 파일 업로드 검증·격리
- [ ] 감사 로그 누락 없음, 개인정보 비저장
- [ ] 레이트 리밋(로그인·발송·AI 질의)

> 보안 민감 변경(인증/인가/개인정보/결제연동)은 머지 전 보안 리뷰 필수. 테스트: [07 §보안](07-testing-strategy.md).

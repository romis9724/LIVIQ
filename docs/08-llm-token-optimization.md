# 08. LLM 토큰 절약 설계

> 아키텍처: [01-architecture.md](01-architecture.md) §RAG/오케스트레이션 · 비용 NFR: [00](00-requirements.md)
> 목표: **품질을 유지하면서 질의당 토큰·비용을 최소화.** 토큰은 1급 설계 제약.

## 1. 비용 구조 (어디서 토큰이 쓰이나)

질의당 비용 ≈ `입력 토큰(시스템+검색컨텍스트+질문) + 출력 토큰`.
RAG는 **검색 컨텍스트가 입력 토큰의 대부분**을 차지 → 여기를 줄이는 게 핵심.

| 비용원 | 절감 레버 |
|--------|-----------|
| 검색 컨텍스트(청크) | 청크 크기·개수·리랭킹·예산 상한 |
| 반복 질의 | 정확/의미 캐시 |
| 시스템 프롬프트 | 프롬프트 캐시(공통 접두부) |
| 고급 모델 남용 | 모델 라우팅(경량/고급) — **보류**([01 ADR-2]) |
| 임베딩 | 증분·중복 방지·배치 |
| 출력 | 간결 응답 지시·max_tokens |

## 2. 캐싱 (가장 큰 효과)

### 2.0 캐시 스코프 (격리 모델 — 최우선)

캐시 키는 `tenant_id`만으로는 부족하다. **접근 스코프**로 키를 분리하지 않으면 같은 단지 내 다른 사용자에게 개인 데이터가 새어 나간다.

| 스코프 | 대상 | 키에 포함 |
|--------|------|-----------|
| `tenant-public` | 공개 문서·FAQ(규약·공지 등 visibility=ALL) | scope + tenant_id + visibility |
| `tenant-role` | 역할 한정 데이터(ADMIN 문서 등) | 위 + role |
| `household-private` | 관리비·평면도 등 세대 단위 | 위 + household_id |
| `user-private` | 본인 민원·개인 대화 | 위 + user_id |

- 키에 **scope + role/visibility + household_id/user_id + 원천 revision**(문서버전·관리비 period·upload_id)을 포함한다.
- **개인 데이터(관리비·민원·개인 대화)는 공유 의미 캐시 금지** — `user-private`/`household-private` 스코프의 **정확 캐시만** 허용.
- **캐시 히트 후에도 요청자의 role·소유권을 재검증**한 뒤 반환한다(캐시는 인가를 대체하지 않음).
- **권한·공개범위 변경 시 관련 스코프 캐시를 무효화**한다(문서 visibility 변경·역할 회수·세대 이동 등).

### 2.1 정확 캐시 (exact)
- 키: **스코프 키(§2.0) + 정규화 질의 + 모델**. Redis.
- 동일 질문 반복(주차/공사시간 등 FAQ) → LLM 호출 0.
- 무효화: 관련 문서 재색인·원천 revision 변경 시 해당 키 만료.

### 2.2 의미 캐시 (semantic)
- **`tenant-public`/`tenant-role` 공개·공유 데이터에만** 적용. 개인·세대 데이터에는 적용하지 않음(§2.0).
- 질의 임베딩 유사도 ≥ 임계치면 캐시 답변 재사용(근거 동일 확인 후).
- FAQ성 반복 질문의 표현 변형 흡수.

### 2.3 프롬프트 캐시
- 공통 시스템 프롬프트·지침을 캐시 가능한 접두부로 고정(공급자 프롬프트 캐싱 활용).
- 변동부(검색 컨텍스트·질문)만 뒤에 배치.

## 3. 검색 컨텍스트 최적화 (RAG)

- **청킹**: 구조 인지(조/항 경계), 토큰 상한 내, **오버랩 최소화**(중복=토큰 낭비).
- **Top-K 축소 + 리랭킹**: 벡터로 넉넉히 가져온 뒤 리랭커로 상위 소수만 LLM에 투입.
- **컨텍스트 예산(`ai-core/budget`)**: 모델 입력 상한의 일정 비율을 검색에 할당, 초과 시 점수순 절단.
- **중복 제거**: 유사 청크 dedupe.
- **메타만 전달**: 표·장문은 요약/발췌만, 원문은 링크로(사용자는 클릭, LLM엔 미투입).
- 근거가 충분하면 추가 청크를 넣지 않음(많을수록 좋다 ✗).

## 4. 모델 구성 (단일 LLM)

| 역할 | 모델 | 비고 |
|------|------|------|
| 생성(전 작업) | **단일 LLM** — OpenAI-호환 엔드포인트(Ollama·vLLM·OpenAI 등), env로 교체 | 선정·교체는 골든셋 비교([07]) |
| 임베딩 | **bge-m3(1024)**, Ollama/vLLM 로컬 실행 | 생성 모델과 별개 고정. 교체=전량 재색인 |

- 단순 분류(의도·긴급도)는 가능하면 **비-LLM**(규칙/소형 분류기)로 대체 — 호출 자체를 없애는 게 최선의 절약.
- 작업별 경량/고급 **모델 라우팅은 보류**: 멀티 모델 운영이 필요하다고 검증될 때 재도입([01 ADR-2]).

## 5. 출력 토큰 제어

- 응답 간결화 지시 + `max_tokens` 상한.
- 구조화가 필요하면 짧은 스키마(불릿/필드) 유도.
- 스트리밍으로 체감 지연↓(비용과 무관하나 UX 보완).

## 6. 임베딩·인제스트 절약

- 문서 해시로 **중복 인제스트 방지**(재업로드 시 재임베딩 X).
- 변경분만 증분 재색인. 배치로 임베딩 호출 묶기.
- 임베딩 모델/차원 고정(변경 시 전량 재색인=대규모 비용, 신중히).

## 7. 비동기·사전계산

- 공지/규약 FAQ는 **사전 생성·캐시**(자주 묻는 질문 미리 답 만들어 저장).
- 문서 색인은 비동기 배치(피크 회피).

## 8. 가드레일 (낭비·남용 차단)

- 질의 레이트 리밋(사용자/단지). 과도 호출 차단.
- 입력 길이 상한. 거대 붙여넣기 거절/요약.
- 무한 재시도 금지(지수 백오프 + 상한).
- 폴백이 LLM 재호출 루프가 되지 않도록(폴백=사람연결/검색결과, 추가 생성 호출 최소).
- **에이전트 스텝 상한**(질의당 도구 호출 2~3회)·도구 결과 크기 상한 — 루프 폭주가 최대 토큰 리스크. 초과 시 현재 근거로 답변/폴백([01 §5.2](01-architecture.md)).

## 9. 측정·예산 관리

- `messages`에 `token_input/output/cost_usd` 기록([03 §4.3]).
- self-hosted(Ollama·vLLM)는 API 과금이 없으므로 비용 축은 **토큰 수 + 지연(GPU 자원)**으로 동일하게 기록·비교.
- 대시보드: 질의당 평균 토큰·비용, 캐시 적중률, 모델별 분포, 단지별 비용.
- **파일럿에서 실제 질의당 비용 측정 → 상한(NFR-COST-01) 확정** → 초과 알림.
- 회귀: 프롬프트·청킹 변경 시 비용 전후 비교(품질과 함께, [07 §5]).

## 10. 우선순위 (효과/노력)

| 레버 | 효과 | 노력 | 도입 |
|------|------|------|------|
| 정확 캐시(FAQ) | ★★★ | 낮음 | MVP |
| 컨텍스트 예산·Top-K 축소 | ★★★ | 중 | MVP |
| 프롬프트 캐시 | ★★ | 낮음 | MVP |
| 의미 캐시 | ★★ | 중 | Phase 2 |
| FAQ 사전생성 | ★★ | 중 | Phase 2 |
| 모델 라우팅(멀티 모델) | ★ | 높음 | 보류 — 필요 검증 후([01 ADR-2]) |
